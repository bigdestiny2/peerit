HiveRelay × Naming, Deploy & Public Content

This guide is for engineers working on the HiveRelay service switchboard and the deploy/name/public-mirror rails that sit around it. The roadmap features land as app-agnostic relay and provider capabilities — so any Pear app can opt into seeding, outboxes, blind custody, name availability, public mirrors, and proofs without custom backend code.

Peerit is not the product that “needs” these features. Peerit is the first reference application built against HiveRelay services — a live conformance fixture that proves an ordinary app can fully utilize the surfaces. PearBrowser is the independent client verifier and discovery surface. Pear Deploy (when built) is the portable compiler/controller that talks to providers through the same capability model.

1. Correct framing

Wrong mental model

  • “Peerit needs ENS and IPFS.”
  • “Ship roadmap features into the social app.”
  • “Each app invents publish/seed/name scripts.”
  • “Relay is Peerit’s backend.”

Right mental model

  • HiveRelay needs portable capabilities, receipts, and service plugins.
  • Pear Deploy needs to orchestrate those providers without owning truth.
  • PearBrowser needs to resolve, verify, and display provenance.
  • Peerit is the first reference consumer proving the surfaces work for any app.

Design test for every roadmap feature

If a feature only works for Peerit, it failed. It must be selectable through the service switchboard (or a signed capability) under an app-neutral contract, with Peerit merely registering a namespace / binding policy like any other consumer.

OutboxLog already states this explicitly in HiveRelay docs: the service is app-neutral; apps sign under a registered namespace (e.g. peerit). Enabling HIVERELAY_OUTBOXLOG=1 without registering namespaces rejects every append. That is the pattern names, IPFS pins, and deploy leases must follow.

2. North star & hard rules

The ecosystem program makes cryptographically owned Pear applications easy to build, preview, promote, name, mirror, observe, and roll forward — while mutable state, private data, authority, and merge policy stay in the native Pear stack and in applications.

Publishers sign meaning → content addresses / logs prove integrity → relays provide availability & transport → clients verify → applications decide trust and merge policy.

HiveRelay’s job

  • Keep Hyperdrive/Hypercore bytes available after publishers go offline.
  • Host optional services (outbox, notify, storage-proof, shards, later names/IPFS).
  • Advertise honest signed capabilities; disappear when unhealthy.
  • Issue receipts, leases, accounting — never ownership claims.
  • Stay blind for private workloads; reject plaintext key material at boundaries.

HiveRelay never becomes

  • A global name registrar or ENS wallet custodian.
  • The source of content truth or merge authority for apps.
  • A mandatory hosted controller for application releases.
  • A public Kubo RPC or unrestricted IPFS pin free-for-all.
  • A place that invents Peerit-specific (or any app-specific) protocol semantics.

Three release loops (keep separate)

3. HiveRelay service switchboard

HiveRelay is a two-layer machine: a kernel (availability, swarm, gateway, custody, accounting) plus a services layer (opt-in providers registered, supervised, catalogued, and dispatched over Protomux / HTTP). Apps do not embed relay internals — they discover capabilities and call methods.

How the switchboard works today

MechanismWhat it doesWhy apps care
ServiceProvider + manifest() Declares name, version, capabilities Discoverable methods without hardcoding peer IPs
ServiceRegistry Register, dispatch, catalog, stats, lifecycle One client API: callService / HTTP facade
Supervision Failed health → remove from catalog; restart budget Broken plugins disappear instead of lying “available”
Signed capability docs Advertise what this relay actually runs Clients/Deploy select by verified capability, not package strings
Opt-in plugins / env flags services.json, HIVERELAY_OUTBOXLOG=1, etc. Operators enable only what they want to host
Namespaces / policies e.g. OutboxLog namespace registration App-agnostic service + app-specific admission

7. Release trains as HiveRelay work

Version slots below follow the ecosystem roadmap. They are planning labels. Each train answers: what HiveRelay must ship, what Deploy/Browser consume, and what reference apps prove.

Foundation — Contract spine

Deploy missing HiveRelay capability freeze needed

Freeze portable contracts, crypto vectors, provider SPI shapes, privacy classifications, evidence predicates, and capability advertising rules before any UI becomes authoritative.

HiveRelay deliverables

  • One signed capability/release manifest as version authority (packages/README/dashboard become projections).
  • Reserve capability names: pear-names-availability-v1, ipfs-bridge-v0 without claiming production policy satisfaction.
  • Freeze request limits (bytes, DAG depth, fanout, redirects, concurrency, time).
  • Evidence schema for benchmark/failure runs consumable by Deploy and CI.
  • Guarantee: sidecar disabled/crashed causes zero regression in native seed/custody/outbox paths.

Reference-app role

Peerit (and later a second app) supply fixtures: clean seed, outbox namespace, cold reader, publisher offline. They do not define the contract.

Release 1 — Verified Previews + Resolve Anywhere

Browser names yes Relay names/IPFS no

Experimental name availability + read-only IPFS sidecar; browser preview resolution of Pear names, .eth, and ipfs:// with provenance and bounded failure.

HiveRelay deliverables

  • pear-names-availability-v1 (opt-in): pin registry core sets; cache raw signed mutable-record envelopes; return all candidate versions + tombstones; signed retention/availability receipts; never assert ownership.
  • Read-only ipfs-bridge-v0 (opt-in): Rainbow or Kubo sidecar; bounded head/get/path/range/CAR; CID verification; explicit disabled/degraded/fetching/available/failed; no pin durability claim.
  • Advertise capabilities only while supervised service is healthy.
  • Kubo RPC loopback-only admin boundary — never app API.

Network enablement

  • Any app’s human-friendly entry points stay resolvable while the owner is offline (availability, not authority).
  • Thin clients open public IPFS content via relay bridge without a second DHT stack.
  • Users distinguish ownership/authenticity from relay availability.

Release 2 — Cryptographic Production

Fleet rails exist Provider SPI TCK incomplete

Provider lifecycle required by Deploy adapters: discover → normalize → quote → compile → prepare → apply → reconcile → observe → verify → renew → revoke. Portable leases, full-request commitments, fencing, exact effect reconciliation.

HiveRelay deliverables

  • Canonical signed/fresh capability + release manifest selection API for Deploy.
  • Full normalized request commitment + effect ID + idempotency + fencing.
  • Provider-signed lease/receipt semantics that distinguish: acceptance vs active lease vs metadata sample vs complete service read vs network completeness vs publisher-offline.
  • Optional sidecars packaged as separately health-checked, resource-bounded artifacts.
  • Keep infrastructure fleet promotion as a separate human-authorized workflow.

Network enablement

  • Any app’s production release can select HiveRelay features by verified capability.
  • Provider replacement/repair without rewriting application artifacts.
  • ENS/IPFS failure cannot invalidate native release integrity or required native availability.

Release 3 — Name and Mirror Anywhere

Not implemented

Authenticated public pinning + optional ENS CCIP gateway as independent binding families. Deterministic public-only mirrors attach to already-valid native releases.

HiveRelay deliverables

  • IPFS pin lifecycle: queued → fetching → pinned → degraded → expired → failed; caller refs; retention; unpin; GC after last valid ref.
  • Receipts that separate desired policy vs observed sidecar state; crash reconciliation.
  • Remote-pinning-compatible surface only where it maps safely into HiveRelay auth + accounting.
  • Start with public immutable data + two independent pin/service sources before “complete storage replica” claims.
  • Optional ens-ccip-gateway-v1: serve publisher-signed offchain Pear records; never private registries; reuse quotas/monitoring/accounting.

Network enablement

  • Any app’s public shell/dataset can dual-publish native Hyperdrive + CID/CAR.
  • Operators sell interoperable public retention using existing receipt models — no new token.
  • Non-Pear IPFS tooling can consume public artifacts without becoming authority for live app state.

Release 4 — Federated Durable App Platform

Multi-op intent exists SLO program incomplete

Multi-operator desired-state reconciliation and repair; independent SLO observers; resource isolation for sidecars; operator UI for pins/expiry/missing blocks/abuse; promote experimental services only with repeatable evidence.

HiveRelay deliverables

  • Multi-operator repair without changing CIDs (mirrors) or rewriting app authority (native).
  • Periodic fresh-reader probes + provider-announcement visibility metrics.
  • Sidecar CPU/memory/fd/storage/connection/bandwidth isolation; native services healthy under IPFS saturation.
  • Suppression/denylist without claiming global content removal on other networks.
  • Multi-relay name consistency observations / optional WitnessLog split views without voting ownership.

Network enablement

  • Independent operators form a measurable public availability network for sites and assets.
  • Apps move/add providers without changing content addresses or protocol semantics.
  • Thin/mobile clients reuse the verified relay boundary without embedding a second P2P stack.

8. Peerit as reference app (not the product)

The roadmap lists Peerit under Release 4 community as a pilot consumer. Read that as: conformance + product pressure, not as ownership of the features.

What Peerit proves for the switchboard

  • OutboxLog can host an ordinary social app with opaque signed rows.
  • Namespaces + roster signing + multi-origin intent are real operator problems.
  • Shell publish + seed + clearnet delivery can coexist with native P2P.
  • Client verification works when relays are untrusted for meaning.
  • Blind shards can be progressive without becoming default public storage.

What Peerit must not become

  • The place ENS/IPFS logic is hard-coded.
  • The de-facto HiveRelay API surface other apps must copy-paste.
  • The owner of OutboxLog wire semantics forever (service stays app-neutral).
  • A reason to delay multi-app TCKs (“works for Peerit” ≠ “works for any app”).

Reference-spec discipline

When Peerit needs a relay behavior, prefer: (1) general service capability, (2) registered namespace / policy, (3) optional plugin — in that order. App-local scripts (publish.mjs, ship reports) are temporary adapters that Deploy should later wrap, then shrink.

How Peerit maps to HiveRelay services today

Private progressive storage blind shard store Any app with ciphertext-off-host needs
Relay roster / failover Signed roster + multi-origin config Provider BindingSet / leases later
Clearnet signed shell Not a HiveRelay service (app release loop) Pear Deploy + independent surface convergence

Strongest ecosystem claim (from the roadmap)

Not “Pear now uses IPFS” or “Pear has decentralized DNS.” Pear apps keep local-first state and cryptographic authority, while the fabric (HiveRelay providers + Deploy + browser verification) makes builds, previews, releases, names, public mirrors, evidence, and providers portable and independently verifiable.

12. Sequencing for HiveRelay engineers

  1. Freeze capability/release authority so fleet strings cannot disagree with signed capability docs.
  2. Harden completeness evidence for seed/pin (metadata + blobs; publisher-offline; fresh reader) — this unblocks honest Deploy predicates.
  3. Ship opt-in names availability as availability-only; parallel with browser provenance UX.
  4. Ship read-only IPFS sidecar with hard bounds and fail-closed incomplete DAGs; native path must ignore sidecar death.
  5. Provider SPI + leases + fencing so Deploy can treat HiveRelay as a portable provider family.
  6. Authenticated pins + optional ENS CCIP only after R2 spine; public-only classification enforced.
  7. Multi-operator repair + SLOs + isolation before availability marketing claims.
  8. Reference apps (Peerit first, then a second independently shaped app) run TCKs — never become the API.

13. Provider & service checklist

Use this when implementing or reviewing a new HiveRelay capability.

  • App-neutral by default
    Admission via namespace/policy/capability — no Peerit-only code paths in the service core.
  • Signed capability advertisement only when healthy
    Supervision removes failed services from catalog and capability docs.
  • Receipts distinguish requested vs observed state
    Acceptance ≠ lease ≠ complete bytes ≠ publisher-offline proof.
  • Kernel survives plugin failure
    IPFS/names sidecars crash without breaking seed/custody/outbox.
  • Privacy classifications enforced at boundary
    Secrets never enter public mirror/name publish paths.
  • Accounting hooks for the resource class
    Bytes, transfer, pin lifetime, routing, repair — portable receipts.
  • Two independent operators in evidence before multi-op claims
    Single-fleet canary is not federation.
  • Reference app + second independently shaped app
    Peerit alone is insufficient for “any application” claims.
  • Infrastructure release loop stays human-gated
    App promote never rolls fleet packages or signs ENS wallets.

14. Non-goals & failure modes

Non-goals

  • Replacing Hyper* with IPFS
  • Embedding Helia in every client
  • CID-as-durability marketing
  • Public Kubo RPC
  • Relay-owned global Pear namespace / .pear TLD
  • ENS key custody on HiveRelay
  • New IPFS token / parallel marketplace by default
  • Deploy as sole app DB / mandatory hosted controller
  • Building roadmap features only for Peerit

Failure modes

  • False availability: seed accepted, blobs incomplete
  • False ownership: relay order decides name winner
  • Authority conflation: ENS/IPFS lag fails native release
  • Capability lie: advertise while service unhealthy
  • App capture: Peerit-shaped wire leaks into “generic” service
  • Sidecar blast radius: IPFS saturation kills native services
  • Privacy leak: public pin of metadata-sensitive content
  • Loop conflation: app ship promotes fleet packages
