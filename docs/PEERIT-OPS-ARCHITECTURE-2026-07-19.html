Peerit Ops Architecture — Standing It Up Fully

Date: 2026-07-19 · App: Peerit 1.0.1 · web release sequence 11 · Goal: full production stand-up on the new blind substrate, served via the HTTPS Hive gateway (T1), with the community moderation v1 feature live and honest.

Peerit is the first real passenger on the HiveRelay giga plane. Keys and verification stay in the browser. Relays and gateways are untrusted availability — they can withhold, never forge. Moderation is a client-honored overlay of signed reports and mod actions, not server-side deletion of truth. Legal takedown on a relay is a separate, identifier-keyed DO-NOT-SERVE veil.

Plane A — Code delivery (T1 gateway)
Static signed Peerit web artifact (HTML/JS/SW). Served by Render today; target composition is publisher Hyperdrive → T1 HTTPS gateway → browser. Gateway must never host private/custody routes.
Plane B — Social data (OutboxLog + Opaque-Log v2)
Signed posts, comments, votes, communities, reports, modactions. Client seals keys to v2!<okey>. Relay stores opaque cells; client materializes feed + moderation.
Plane C — Optional threshold custody (blind cells)
Large bodies / sealed blobs via blind-shard dealer when over threshold. Not required for every post. T2 custody vaults MUST NOT run the public app gateway.
Forbidden claims (copy discipline):
  • Never: “the operator can’t read public posts” — public read key reaches every reader, including the operator.
  • Never: “moderation deletes the record from the network” — records stay signed; visibility is local policy.
  • Never: “one gateway outage erases Peerit” — if multi-gateway + multi-outbox are proven; today’s canary is single-ingress.
  • Never: “gateway is a custody vault” — T1 ≠ T2.
Allowed claim: “In the ordinary course, the operator holds and serves nothing readable and must affirmatively reconstruct to read.”

3. Community moderation — what we built

Moderation is a first-class product surface, not a server flag. Relays store signed opaque cells; every client runs the same pure policy over its admitted local view (js/moderation.js). Policy is orthogonal to ranking.

3.1 Record types

TypeSemanticEffect
report Signed LWW slot per community×target×author · verdict bury / keep Drives consensus thresholds; sealed in Opaque-Log v2
modaction Content: remove, approve, lock, unlock, sticky, unsticky
User: ban, unban, addmod, removemod		 Moderator overlay; creator always trust root
Withdrawals Signed tombstones in the same report slot Fully reversible burial path

3.2 Visibility ladder & thresholds (v1 constants)

StateDefault thresholdReader experience
visible—Normal feed row
downranked≥3 bury · support ≥ 0.60Lower rank / soft friction
collapsed≥5 bury · support ≥ 2/3Collapsed with reveal
buried≥7 bury · support ≥ 0.75Buried; “show anyway” session exception

3.3 Who may report (anti-Sybil roots)

  • Trust roots: community founder + current moderators (from signed addmod chain).
  • Eligible reporters: non-banned members who published content in-community and received a direct positive vote from a root.
  • Explicitly not used: global reputation, recursive upvote graphs, backdatable account age.

LayerWhat operators getWhat they do not get
Opaque-Log v2 (on) No passive semantic key grep; content-neutral cells; drop-by-opaque-id possible Confidentiality of public posts; anonymity of authors; free prefix “list r/x”
OutboxLog transport Server-side Ed25519 verify; ecosystem-standard /api surface Blindness by itself — blindness is 100% client seal
HiveRelay blind namespace For private/capability content only Usable for public forum content (read key is universal → forbidden)
Blind cells / shards Threshold custody for large sealed bodies Automatic for every short post; public-read secrecy

Ops implication: stand Peerit on HiveRelay OutboxLog in the non-blind peerit namespace with client-side v2 sealing. Do not put public forum records into a blind-only namespace that pretends the operator cannot reconstruct.

RuleOps meaning
Application key is canonical Every HTTPS name is a mirror of a Hyperdrive / signed release — not an identity root
Exact-byte serve Gateway must not mutate app bytes in verifiable mode
T1 vs T2 split Public gateway on T1 only; blind custody vaults never mount app-hosting routes
Multi-gateway Target: unrelated gateway operators; today’s peerit.site is still Render-centric canary
Phase-1 evidence G1–G6 executed on v1-integration (2026-07-18); G7–G13 need real operators / digests / host ceilings
Near-term hybrid (honest): keep serving the signed web sequence from peerit.site (Render) while the T1 gateway canary is proven in parallel. Cut over code delivery to gateway only after G7–G13 + dual-gateway failover proof. Data plane cutover to multi-HiveRelay OutboxLog is independent of code-delivery cutover.

8. Operator checklist

8.1 Today’s canary (keep green)

  • Signed web sequence verifiable (npm run web:verify)
  • Writable atomic commit path live on single ingress
  • Opaque-Log v2 write/read path live
  • Community moderation v1 in client + unit tests
  • Apply deploy/render-security-headers.json (CSP etc.) and re-proof
  • Link Render blueprint to existing peerit-site service
  • Off-host backup of outbox journal (today: same-host timer)

8.2 Blind + HiveRelay stand-up

  • Provision ≥1 production HiveRelay with OutboxLog + Peerit namespace admission
  • Build web with --relay-backend hiverelay-outbox --relay https://…
  • Confirm boot probe / manual /api/bridge/status → service: outboxlog
  • Sign new web sequence (bump releaseSequence) with new roster
  • Run proof:hiverelay-outboxlog + live moderation smoke (report → bury → view Open)
  • Implement DO-NOT-SERVE per RELAY-TAKEDOWN-SPEC.md; drill opaque-id suppress
  • Add second operator; clear singleIngressWriter; failure-injection proof

8.3 HTTPS gateway stand-up

  • Keep T2 custody nodes gateway-free
  • Publish Peerit app on T1 gateway canary with exact-byte verification
  • Complete G7–G13 (two operators, digests, Docker nginx capture, host ceiling)
  • Optional: map publisher domain → gateway with lease; dual-gateway failover test
  • Claim lint: no “private content via gateway”

8.4 Moderation go-live criteria

  • Readers can switch Community / Consensus / Open
  • Reports sealed in v2 cells; index aggregates without relay semantics
  • Mod tools for remove/lock/sticky/ban/addmod in UI
  • Shadow-mode metrics on thresholds before tightening defaults
  • Document appeals / “show buried” UX for operators and communities
  • Legal: operator uses DO-NOT-SERVE, not “edit the mod overlay,” for notices

9. Failure & liability model

EventExpected behaviourWho acts
Community spam wave Eligible bury quorum collapses/buries; Open readers still see if they choose Community + client policy
Moderator removes post Overlay hides for Community view; signed post still in outboxes Mod keys
Valid legal notice to operator Suppress opaque id / appId on this relay only; other peers may retain Relay operator
Single outbox host down Canary degraded; multi-ingress target continues writes Ops failover
Gateway down Code delivery fails until mirror/gateway #2; data plane independent if already loaded / SW pinned Publisher + gateway ops
Malicious relay forges records Client reject at verify — never merge Automatic
