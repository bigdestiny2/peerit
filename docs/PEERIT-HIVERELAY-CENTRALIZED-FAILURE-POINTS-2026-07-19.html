Centralized Points of Failure

Date: 2026-07-19 · Scope: What remains concentrated when the design is fully executed — multi-ingress OutboxLog, T1 HTTPS gateway cutover, Opaque-Log v2, DO-NOT-SERVE, moderation v1, optional blind cells, Peerit as first app.

Even after “full stand-up,” the system removes “one VPS owns the truth” more than it removes “a small set of keys and operators own the rails.” Some points are true single points of failure (SPOFs); others are shared dependencies that fail a class of users even if the network as a whole is not one box.

1. Severity legend

ClassMeaning
Hard SPOF One key, service, or party whose loss/compromise breaks a whole generation of the system for most users.
Concentration Multiple parties possible, but discovery, policy, or law clusters around few operators or rails.
Social / product Not network-global, but captures default UX or community power.
By design Intentionally not global; can still look “central” if many conduits act in lockstep.

3. Hard SPOFs — trust anchors

3.1 Network read key (PEERIT_READ_KEY) Hard SPOF

What it is: One app-wide HMAC key that derives all Opaque-Log v2 okeys and unlocks public sealed envelopes.

Why it concentrates risk

  • Rotation ≈ migration. The blind-outbox design treats RK as non-rotatable without a clean cutover. Compromise or loss is existential for that generation of cells.
  • Not confidentiality vs operators. Every client (including any operator-run client) holds RK. Blindness is “no passive grep at rest,” not “operator can’t reconstruct.”
  • Forcing function: If RK leaks, an adversary can confirm and enumerate more cheaply offline once they have cell dumps — not forge signatures, but collapse the liability posture you paid for.

Failure modes: Key compromise → opaque-id / “ordinary course” story collapses for public content. Key loss → new clients can’t open the corpus without a coordinated reseal.

3.2 Web release signing key + sequence authority Hard SPOF

What it is: Offline Ed25519 that signs asset-manifest / web sequence; clients pin pinnedReleaseKey.

Why

  • Whoever holds the release seed defines “official Peerit code” for normal browsers (SRI/SW pin what that key signed).
  • Sequence floor logic rejects rollbacks/forks of the same sequence with different bytes — lost/compromised key is a code-delivery CA problem.

Failure modes: Compromised release key → malicious web bundle that still “verifies.” Lost key → no honest upgrades without pin ceremony / hard fork of trust anchors.

Partial escape hatch: PearBrowser host path; pure peerit.site users ride this rail fully.

3.3 Relay roster signing key + bootstrap list Hard SPOF

What it is: Signed roster of outbox/gateway endpoints; canary still starts from bootstrap URLs.

Why

  • Roster key is the discovery CA for “which HTTPS outboxes are allowed.”
  • Even with multiple relays, clients only trust roster-admitted ones. One roster signer gates which operators count.
  • singleIngressWriter: true (canary) is a hard write SPOF; after multi-ingress, admission policy remains centralized unless roster authority is federated.

Failure modes: Bad roster → clients steered to hostile/withholding operators. No rotation process → stuck on dead endpoints. Legal pressure on roster holder → global deplatform of discovery, not of peer copies.

3.4 Writer identity = outbox appId (Ed25519) Hard SPOF (per user)

What it is: Each user’s signing key is their outbox identity; recovery is phrase/export-dependent.

Why

  • No network can restore a lost private key without a recovery design that is itself central or social.
  • Compromise = full impersonation (posts, votes, modactions, reports) until peers/social process stop trusting that pubkey.

Failure modes: Key loss → account death. Key theft → forged social graph until ban/social process.

4. Operational concentration

4.1 OutboxLog namespaces & operator admission Concentration

What it is: HiveRelay namespaces are operator-admitted; Peerit runs in a named namespace with policy (caps, blind mode rules, takedown).

Why

  • Multi-relay only helps if several independent operators admit the same app namespace and honour the same wire rules.
  • Namespace config, rate limits, bytesPerDay, and DO-NOT-SERVE are still per-operator — coordination is social/legal, not cryptographic consensus.

Failure modes: Coordinated refusal or config divergence → split brain (different suppress sets, different caps). No automatic global consistency of takedowns (by design).

4.2 T1 HTTPS gateway + DNS / WebPKI Concentration

What it is: Exact-byte public Hyperdrive serve + TLS names (peerit.site, *.hive.<gateway>, etc.).

Why

  • Browser users must fetch code somehow. Multi-gateway still depends on DNS + CA for first load (and often updates), unless already-pinned SW / Pear / IPFS/ENS mirrors.
  • Gateway operators see IP, path, timing of public fetches.
  • Spec forbids treating gateway as identity — but it remains a high-leverage choke for availability of the app binary.

Failure modes: Domain seizure, TLS mis-issue, gateway block, CDN policy → large fraction of browser users can’t boot. Data plane may still work for clients with SW pin / Pear install.

Mitigations (not free): multiple gateways, publisher domain leases, IPFS/Arweave mirrors, PearBrowser native install. Fully executed still has bootstrap concentration.

4.3 Blind cells / k-of-n custody Concentration

What it is: Optional large-body path: secrets split across vaults.

Why

  • Still needs a roster of vault operators. Small k or co-located vaults (jurisdiction, hoster, identity) → collusion or correlated outage.
  • Placement/repair policy is an org/process concentration even if crypto is distributed.

Failure modes: < k online → unrecoverable sealed body. ≥ k collude → reconstruct without user. One cloud region → correlated loss.

5. Social / product concentration

5.1 Moderation trust roots (founder + mod chain) Social

What it is: Client overlay; founder always a mod; addmod/removemod chain; report eligibility tied to roots.

Why

  • Not global network centralization, but per-community centralization of effective visibility for Community / Consensus views.
  • Open view reduces this; default UX still amplifies root power.
  • Sybil-resistant eligibility is local — roots can still capture a small community.

Failure modes: Hostile founder/mod set → mass bury/remove for Community viewers. Does not delete bytes network-wide, but can nuke the default feed experience.

5.2 Client policy constants (thresholds, PoW, rankers) Social

What it is: Bury thresholds, PoW bits, built-in algorithms shipped in the signed web build.

Why: Changing “what consensus means” is a release-key decision, not an on-protocol vote of users. Policy evolution stays with whoever ships signed sequences (or Pear catalogue pins).

Failure modes: Bad threshold update → mass bury or spam win; old SW pins disagree with new clients → social split.

6. By design: DO-NOT-SERVE

6.1 Per-conduit suppression By design

What it is: Operator suppress list by opaque id / appId (serve-time veil; bytes retained).

Why (design honesty)

  • Intentionally not global erasure — correct conduit posture; other peers/seeders may retain content.
  • Legal compliance is therefore fragmented.
  • Conversely, if all major ingress operators apply the same suppress set (roster-coordinated or shared legal blast radius), you get de facto centralized unavailability for browser users who only use those ingresses.

Failure modes: Single-operator suppress → local withholding only. Industry-wide notice → multi-ingress disappearance for web path while Pear/seeders retain copies.

7. What is not a central SPOF (when fully executed)

Decentralized enough (honest list)

  • Content authenticity — no relay can forge Ed25519 records; merge rejects forgeries.
  • Single relay availability — multi-ingress + peer replication can survive one outbox host if clients use multiple and data has been replicated.
  • Gateway as source of truth for posts — posts live in outboxes/peers; gateway is T1 code/media distribution.
  • Moderation as global delete — Open view + other peers still hold signed records; moderation is not network-wide erase.

9. Why this shape is intentional

The architecture optimises for:

  1. Unforgeable social data without trusting relays.
  2. Operator as conduit (withhold/takedown by opaque id) rather than as curator of meaning.
  3. Honest privacy ladders (no fake “operator can’t read public forums”).

The cost is that bootstrap, discovery, and policy evolution stay relatively central (keys + signed releases + roster), while storage and replication can be plural.

Fully executed, you should measure success as: no single VPS can rewrite history — not no one can take the web product offline or redefine the official client.

10. Implications for operator dashboard & stand-up

  • Dashboard should surface SPOF-adjacent health (disk, sole-writer, admin armed, backup age, Tor degraded) without displaying content — see HIVERELAY-OPERATOR-DASHBOARD-DESIGN-2026-07-19.html.
  • Ops stand-up should explicitly track multi-ingress and roster independence as first-class gates, not afterthoughts after “features ship.”
  • Product copy must never claim RK or multi-ingress removes what only federated discovery + offline roots remove.
